# CookRag 需求文档（仅需求范围）

> 本文档只描述“做什么”和“验收标准”，不包含技术实现细节。  
> 架构、数据存储、接口实现方案请查看 `docs/backend-tech.md`。

## 1. 背景与目标

CookRag 是一个做饭网站。用户在网页输入中文烹饪需求（如“麻婆豆腐”“辣”“家常茄子”），系统返回相关中文菜谱并支持查看详情。

项目目标：
- 提供稳定的中文菜谱搜索体验。
- 支持用户账号体系（注册、登录）。
- 支持点赞互动并在结果中体现热度影响。

## 2. 业务范围

### 2.1 本期必须实现
- 菜谱导入：支持从 `recipes/` 中导入 Markdown 菜谱。
- 菜谱搜索：根据用户中文输入返回相关菜谱列表。
- 菜谱详情：展示食材、步骤、标签、烹饪时长等信息。
- 用户系统：注册、登录。
- 点赞系统：登录用户可点赞/取消点赞，并可看到点赞数。
- 前端可用：搜索页与详情页可正常使用。

### 2.2 Out of Scope（本期不做）
- 多语言翻译与跨语言检索。
- 收藏、评论等社交功能（除点赞外）。
- 计时器、营养分析等扩展工具。

## 3. 用户故事

1. 作为用户，我能输入中文菜名/口味/食材偏好，并得到相关菜谱。
2. 作为用户，我能进入菜谱详情，看到食材与步骤。
3. 作为用户，我能注册并登录账号。
4. 作为登录用户，我能点赞/取消点赞并看到点赞数变化。
5. 作为用户，我希望结果同时考虑“相关性 + 受欢迎程度”。
6. 作为维护者，我能重复执行导入，且不会产生重复脏数据。

## 4. 输入与内容约束

### 4.1 查询输入
- 仅支持中文输入。
- 非中文输入需要有明确且一致的处理方式（拦截或错误提示）。

### 4.2 菜谱数据
- 菜谱来源为中文 Markdown 文件。
- 文件格式允许不完全一致，但应尽可能提取结构化字段（标题、食材、步骤等）。

## 5. 功能需求（产品视角）

### 5.1 搜索
- 用户提交查询后，系统返回不超过 `topK` 的候选列表。
- 每条结果至少包含：菜谱标识、标题、相关度分数、点赞数、综合排序分数。
- 返回结果按综合排序分数降序。

### 5.2 详情
- 用户可通过菜谱标识获取详情。
- 不存在的菜谱标识返回“未找到”语义。

### 5.3 用户注册与登录
- 注册：昵称必填，密码需满足最小长度要求。
- 登录：账号密码正确则登录成功并返回身份凭证。
- 昵称重复应返回冲突语义错误。
- 密码错误应返回鉴权失败语义错误。

### 5.4 点赞
- 仅登录用户可点赞/取消点赞。
- 点赞与取消点赞均需幂等。
- 任意时刻点赞数不能为负数。

### 5.5 导入
- 递归读取 `recipes/**/*.md`。
- 导入脚本可重复运行，且不应重复创建同一菜谱。

## 6. 非功能需求（NFR）

### 6.1 性能
- 在常见数据规模下，搜索应保持可交互响应速度。

### 6.2 可用性
- 系统重启后仍可继续搜索与查看详情。
- 导入流程失败时应有可定位问题的错误信息。

### 6.3 安全
- 用户密码不得以明文存储。
- 接口访问应具备基础鉴权与跨域访问控制。

### 6.4 可观测性
- 应记录关键业务日志：导入进度、搜索失败、鉴权失败、点赞失败。

## 7. 验收标准（Acceptance Criteria）

1. 导入验收
   - `recipes/` 中至少 3 个 Markdown 菜谱可成功导入。
   - 子目录中的文件也能被导入。
   - 重复导入不会造成同一菜谱重复创建。

2. 搜索验收
   - 中文查询可返回语义相关结果。
   - `topK` 参数生效，返回条数不超过 `topK`。
   - 每条结果包含相关度、点赞数、综合分数，且按综合分数降序。

3. 详情验收
   - 已存在菜谱可返回完整详情字段。
   - 不存在菜谱返回未找到语义错误。

4. 账号验收
   - 注册、登录流程可用。
   - 昵称重复返回冲突语义错误。
   - 密码错误返回鉴权失败语义错误。

5. 点赞验收
   - 登录后可点赞与取消点赞。
   - 重复点赞不重复累计；重复取消不产生负数。

6. 前端联调验收
   - 搜索页能展示结果列表。
   - 详情页可展示关键字段。
   - 点赞数在交互后可观察到变化（实时或刷新后）。
